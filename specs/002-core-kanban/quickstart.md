# Quickstart: Validating Core Kanban

**Feature**: `002-core-kanban`

Walks through validating each user story's acceptance scenarios end-to-end. This is a
validation guide, not an implementation guide — see `tasks.md` (generated by
`/speckit-tasks`) for the build steps themselves.

## Prerequisites

- Feature 001 complete: local dev environment running, database migrated, at least
  one authenticated account (Google, GitHub, or credentials).
- Schema migration for this feature applied (`Board.taskCounter`, `Task.code`
  required, `Task.sprintId`, `Invitation.expiresAt`, `ActivityEvent.actorId` — see
  `data-model.md`).
- Two browser sessions (or one normal + one incognito) signed in as two different
  users, to exercise multi-member scenarios (invite, assign, concurrent creation).

```bash
pnpm prisma migrate dev --name core-kanban-fields
pnpm dev
```

## US3 — Workspace and board setup (foundation for everything below)

1. Sign in, create a workspace with a name → confirm it appears with a generated
   slug and you're its `OWNER`.
2. Invite a second email with role `MEMBER` → confirm the invitation shows
   `PENDING`; accept it from the invited account → confirm the member now appears
   with role `MEMBER` and can sign in and see the workspace.
3. Create a board, pick the `KANBAN` template → confirm default columns appear.
4. Add a column, rename it, recolor it, reorder it relative to the others → confirm
   each change is reflected immediately.

**Expected outcome**: SC-001 (workspace → usable board in under 3 minutes).

## US1 — Fluid task board interaction

1. On the board from US3, create a task in the first column → confirm it appears
   immediately with a code like `AX-0001`.
2. Drag the task to the second column → confirm the move is visible before any
   network delay would allow a server response (SC-002).
3. Throttle the network (browser devtools) and repeat a drag that will fail →
   confirm the card returns cleanly to its original position, no stuck spinner, no
   duplicate card.
4. Create several tasks in one column, drag one to a middle position → reload the
   page → confirm the order persisted.
5. From two sessions, create a task on the same board within the same second in
   both → confirm both tasks exist with different codes (SC-003; for a stronger
   check, script 10 concurrent `createTask` calls against the same board and assert
   10 distinct codes).
6. Delete a task → confirm it's gone immediately and stays gone after reload.

## US2 — Rich task detail

1. Open a task's detail view → edit description (bold/list/link), priority,
   estimate, due date → confirm each saves without a page reload.
2. Assign the second workspace member → confirm they appear on both the detail view
   and the board card.
3. Add and remove a label → confirm both surfaces update.
4. Post a comment → confirm it appears with author and timestamp.
5. Open the activity history → confirm every change made in steps 1–4 appears, in
   order, correctly attributed (SC-006).

## US4 — Role-based permission enforcement

1. Set the second member's role to `VIEWER`.
2. As that `VIEWER`, load the board → confirm full read access (tasks, details,
   comments, activity all visible).
3. As that `VIEWER`, confirm mutating controls (new task, drag, edit, delete,
   invite) are unavailable in the UI.
4. As that `VIEWER`, call a mutation Server Action directly (e.g. via a script or
   browser console invoking the action with the `VIEWER` session's cookies) for at
   least: create task, move task, edit workspace name, invite a member → confirm
   every one is rejected server-side (`INSUFFICIENT_ROLE`) regardless of the UI
   (SC-004).
5. As a `MEMBER`, attempt to delete the workspace or change another member's role →
   confirm rejection.

## US5 — Sprint planning

1. On a `SCRUM`-template board, create a sprint with a name and a valid date range →
   confirm it appears as `PLANNED`.
2. Attempt to create a sprint with `endDate` before `startDate` → confirm rejection
   with a clear validation message (FR-014).
3. Attach two existing tasks to the sprint → confirm both show as part of it.
4. Change the sprint's status `PLANNED` → `ACTIVE` → `COMPLETED` → confirm each
   transition is reflected wherever the sprint is shown.

## Edge cases to spot-check

- Try deleting a column that still has tasks → confirm it's blocked with guidance to
  move/remove them first.
- Remove a workspace member who is assigned to an open task → confirm the task is
  unassigned from them but their existing comments/activity entries remain intact.
- View a board with zero columns/tasks → confirm the empty state (from
  `axiom-design/axiom_empty_board_state/`) renders instead of a blank/broken layout.
- Invite an email that's already a member → confirm rejection, no duplicate
  membership or invitation created.
