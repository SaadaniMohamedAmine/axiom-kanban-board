# Quickstart: Validating Project Setup & Foundation

**Feature**: `001-project-setup`

This walks through validating each user story's acceptance scenarios end-to-end. It is
a validation guide, not an implementation guide — see `tasks.md` (generated by
`/speckit-tasks`) for the build steps themselves.

## Prerequisites

- Node.js + pnpm installed locally.
- A Vercel Postgres database provisioned (see `research.md` §1), its connection string
  available as `DATABASE_URL`.
- OAuth app credentials created for Google and GitHub (redirect URLs pointed at the
  local dev URL and the production Vercel URL).
- A Pusher Channels app (free tier) with its app id/key/secret/cluster.
- `.env.local` populated with all of the above plus placeholders for `GROQ_API_KEY` /
  `GEMINI_API_KEY` (unused this phase, provisioned ahead of time per FR-007).
- A Vercel project linked to the Git repository, with the same variables mirrored into
  its environment settings.

## US1 — Local environment boots and builds

```bash
pnpm install
pnpm dev      # visit the local URL — a page must render with no console errors
pnpm build    # must exit 0
```

**Expected outcome**: dev server serves a page; production build completes with zero errors (SC-001).

## US2 — Database schema matches the data model

```bash
pnpm prisma migrate dev --name init
pnpm prisma studio   # or any query — confirm every table in data-model.md exists
```

**Expected outcome**: every entity listed in `data-model.md` exists as a table; a query
against the production `DATABASE_URL` succeeds from a deployed function (SC-002).

## US3 — Account creation without email verification

Manually, on the running app's sign-up screen:

1. Sign up with Google → lands authenticated, no verification prompt.
2. Sign up with GitHub (different email) → lands authenticated, no verification prompt.
3. Sign up with a new email/password → lands authenticated, no confirmation email.
4. Attempt to sign up again with an email already used in step 1, via a different
   method (e.g. email/password) → expect the `409` conflict response and inline error
   from `contracts/auth-conflict.md`, and confirm no new `User`/`Account` row was
   created for that attempt.

**Expected outcome**: steps 1–3 succeed with zero verification friction (SC-003); step 4 is blocked as specified.

## US4 — Realtime channel round-trip

Following `contracts/realtime-test-channel.md`: open a client subscribed to
`setup-test`, trigger a `ping` publish, and measure the time to receipt.

**Expected outcome**: event received in under 1 second (SC-004).

## US5 — Live production URL

```bash
git push                 # triggers Vercel's automatic deploy
```

Then, from a device outside the local network, visit the production URL.

**Expected outcome**: the login/sign-up screen renders publicly (SC-005). Separately,
run a secret-scan (e.g. `git log -p | grep`-style check for known key patterns, or a
dedicated secret-scanning tool) across the repository history before the first public
push, per FR-007 and the constitution's Security & Scope Integrity principle.

## Accessibility check (SC-006)

On the sign-up screen: tab through every interactive element (focus visible throughout,
no keyboard trap) and verify text contrast against `axiom-design/axiom/DESIGN.md`'s
documented palette meets 4.5:1.
